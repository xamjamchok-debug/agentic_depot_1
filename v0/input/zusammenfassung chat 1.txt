

Ich möchte in diesem neuen Chat den bisherigen Stand meines Custom-GPT-Projekts zur Depotanalyse konsistent prüfen und technisch bereinigen. Bitte lies alles gründlich, arbeite systematisch und prüfe auf Konsistenz zwischen GPT-Hinweisen, Action-Schema, Apps-Script und gewünschtem Verhalten.

KONTEXT UND ZIEL

Ich baue ein Custom GPT für Depotanalyse. Ziel ist:
1. Depot-Screenshots und Dateien (DKB, comdirect, CSV/Excel/Umsatzlisten) analysieren
2. strukturierte Reports erzeugen
3. Daten optional in ein Google Sheet persistieren
4. später historische Vergleiche, Deltas, Risikoanalysen und Zukauf-/Verkaufserkennung ermöglichen

Das System soll:
- DKB-Screenshots (dark Investments-Liste) verarbeiten
- comdirect-Screenshots / Tabellenansichten verarbeiten
- Cashflow-/Umsatzlisten verarbeiten
- Positionen, Performance, Aggregationen, Risiken und Treiberanalysen liefern
- gespeicherte Snapshots + Positionsdaten + Cashflows in Google Sheets schreiben
- gespeicherte Daten wieder abrufen können

WICHTIGER HINWEIS ZUR ARCHITEKTUR

Es gibt drei Ebenen:
1. GPT-Hinweise / GPT-Definition
   - Textinstruktionen für das GPT
2. GPT Action Schema
   - YAML OpenAPI Schema in ChatGPT Actions
3. Google Apps Script
   - JavaScript im Google Sheet (Apps Script Web App)

Zur Laufzeit erzeugt das GPT JSON-Payloads und schickt sie an die API.

AKTUELLE KERNARCHITEKTUR

Google Sheet:
- 1 Spreadsheet
- 3 Tabs:
  - snapshots
  - positions
  - cashflows

Header-Struktur:

Tab snapshots:
snapshot_id | timestamp_berlin | broker | depot_id | portfolio_value_eur | notes

Tab positions:
snapshot_id | instrument_key | isin | wkn | name_raw | asset_class | region | sector | quantity | price | market_value_eur | unreal_pl_eur | unreal_pl_pct

Tab cashflows:
date | broker | amount_eur | type | instrument_key | memo_raw

Google Apps Script läuft als Web App.
Das GPT ruft die Apps-Script-Web-App über Actions auf.

DERZEIT GEWÜNSCHTES VERHALTEN DES SYSTEMS

A) Snapshot-Analyse
- Depot-Screenshots analysieren
- Positionen extrahieren
- ggf. CSV-Daten matchen
- Report erzeugen
- nur bei explizitem Befehl speichern

B) Snapshot-Speichern
- Action: append_snapshot
- schreibt snapshots + positions

C) Cashflow-Analyse
- Cashflow-/Umsatzliste analysieren
- depotrelevante Cashflows erkennen oder je nach Instruktion alle Kontobewegungen
- nur bei explizitem Befehl speichern

D) Cashflow-Speichern
- Action: append_cashflows
- schreibt cashflows

E) Read / History
- letzter Snapshot + zugehörige Positionsdaten abrufen
- Action: get_snapshot_bundle

F) weitere History-/Vergleichslogik
- geplant / teilweise rudimentär

WICHTIGE FACHLICHE ANFORDERUNGEN

1. Das GPT soll standardmäßig NICHT automatisch speichern.
2. Speichern nur bei expliziten Befehlen wie:
   - speichern
   - analysieren und speichern
   - diesen Snapshot speichern
   - ins Google Sheet schreiben
3. Historie abrufen nur bei explizitem Befehl.
4. Fehlende Daten nicht erfinden.
5. Zeitstempel ist Pflicht für jeden Snapshot.
6. EUR ist Standardwährung.
7. Zukäufe / Verkäufe sollen erkannt werden:
   - primär über Cashflows
   - fallback über Snapshot-Vergleich (steigende Stückzahl = Zukauf, sinkende = Verkauf)

WICHTIGE BEOBACHTETE PROBLEME IM BISHERIGEN VERLAUF

1. GPT hat anfangs beim Speichern falsche JSON-Feldnamen erzeugt, z. B.:
   - timestamp statt timestamp_berlin
   - total_value_eur statt portfolio_value_eur
   - name statt name_raw
   - value statt market_value_eur

2. GPT hat bei Cashflow-Dateien zunächst fälschlich append_snapshot statt append_cashflows verwendet.

3. OpenAPI-Schema war anfangs fehlerhaft / unpräzise:
   - parse error
   - payload zu generisch
   - Action-Parameterfehler
   - Schema-/Parameterfehler (payload)

4. Cashflow-Speicherung schlug mehrfach fehl, obwohl das GPT bereits cashflows extrahiert hatte.
   Das deutete darauf hin, dass:
   - das Action-Schema payload.cashflows nicht präzise genug beschrieben hat
   - oder Apps Script zu fragil war

5. GPT extrahierte z. B. nur 19 Cashflows statt ~200 Kontobewegungen.
   Fachliche Interpretation:
   - wahrscheinlich nur depotrelevante Buchungen extrahiert
   - nicht alle Girokonto-Buchungen
   Das ist fachlich möglich, muss aber transparent und steuerbar sein.

6. Performance-Probleme:
   - Speichern von 100–200 Zeilen dauerte teils mehrere Minuten
   - Verdacht: appendRow in Schleife statt Batch-Write per setValues

7. Positionsdaten:
   - Write-Seite grundsätzlich vorhanden über append_snapshot
   - Read-Seite zunächst unvollständig
   - deshalb neue Read-Action get_snapshot_bundle ergänzt

AKTUELLER STAND DER ACTIONS / API-LOGIK (KONZEPT)

Es sollen derzeit mindestens diese Actions/Action-Types existieren:
- append_snapshot
- append_cashflows
- get_snapshot_bundle
- get_history

SEMANTIK DER ACTIONS

1. append_snapshot
Input:
{
  "action": "append_snapshot",
  "payload": {
    "snapshot": {
      "snapshot_id": "...",
      "timestamp_berlin": "...",
      "broker": "dkb|comdirect",
      "depot_id": "...",
      "portfolio_value_eur": 12345,
      "notes": "..."
    },
    "positions": [
      {
        "instrument_key": "...",
        "isin": "...",
        "wkn": "...",
        "name_raw": "...",
        "asset_class": "...",
        "region": "...",
        "sector": "...",
        "quantity": 0,
        "price": 0,
        "market_value_eur": 0,
        "unreal_pl_eur": 0,
        "unreal_pl_pct": 0
      }
    ]
  }
}

2. append_cashflows
Input idealerweise:
{
  "action": "append_cashflows",
  "payload": {
    "cashflows": [
      {
        "date": "...",
        "broker": "...",
        "amount_eur": 0,
        "type": "buy|sell|dividend|fee|tax|deposit|withdrawal|unknown",
        "instrument_key": "...",
        "memo_raw": "..."
      }
    ]
  }
}

Wegen GPT-Instabilität wurde die Idee verfolgt, Apps Script robuster zu machen und nicht nur payload.cashflows zu akzeptieren, sondern ggf. auch rows/entries/items und alternative Felder:
- amount_eur oder amount
- memo_raw oder memo oder description
- instrument_key oder isin oder wkn oder name

3. get_snapshot_bundle
Input:
{
  "action": "get_snapshot_bundle",
  "payload": {
    "snapshot_id": "<optional>",
    "broker": "<optional>",
    "depot_id": "<optional>"
  }
}
Verhalten:
- wenn snapshot_id vorhanden: genau diesen Snapshot laden
- sonst: letzten Snapshot für broker/depot laden
- Rückgabe:
  - snapshot
  - positions

4. get_history
Derzeit eher rudimentär.
Bitte prüfen, ob diese Action aktuell sinnvoll, konsistent und ausreichend ist.
